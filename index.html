对象池模式

对象池模式是一种 creational 设计模式。该模式下，池中所包含的对象将不再是每次需要时创建，使用完毕后销毁，而是初始化好后放入池中等待使用。使用该池的 client 每次需要时都会向对象池请求一个对象，在上面进行一些操作，使用完之后返回给对象池。这个过程可以是手动执行或自动执行。使用对象池的目的主要是为了提升性能，尤其是当需要和一些创建昂贵但使用时间不长的对象打交道时。

初始状态下，池模式创建了一个可复用的初始化好的对象的集合。当 client 需要一个新对象时，它会向对象池发起请求，如果已经有可使用的对象，则立刻返回一个对象。如果当前池中没有可使用的对象，则会创建一个然后返回。一些对象池使用的资源型的对象会有最大个数限制，如果达到个数后有新的请求时，可以选择抛出异常，或 block 当前请求的线程直到有对象可使用，也可以提升对象池的限制数。

对象池模式的常见使用场景是数据库连接池、网络连接池和线程池等，也可以用在使用第三方资源的地方。

注意事项：

  • 如果创建对象的代价不昂贵，则无需使用。否则反而浪费内存。
  • 使用完毕后返回对象时，不应该存在对它的引用。
  • 返回的对象应该清除自身的状态，否则可能会引起几个问题，一个是上一个 client 的一些状态（比如认证信息等）给了下一个 client，还有就是上一个 client 的一些隐私数据泄露给了下一个 client。
  • 一定要考虑线程安全。

实现

对象池在 C++ 中可以用过 smart pointer 实现。在有 GC 的语言中，由于没有 destructor 的存在，所以要手动实现。不建议使用 finalizer 来归还对象，因为不能确保它什么时候执行。可以使用 using 和 try...finnally 来保证即使抛出异常了也能正常归还对象。

单例模式

单例模式是一种 creational 设计模式，用来确保一个类只有一个实例，并提供一个全局访问点。由于不少类由于特殊性，需要在整个程序运行中保持只有一个实例，比如注册表和连接池等。为了保证一个类只被实例化一次，并且该实例可以被多次引用，便有了单例模式。

核心思想

将类的构造函数私有化，对外只开放一个静态静法作为唯一入口。通过该入口判断是否已有该类的对象，如果有，则返回现有对象，否则在类中调用私有的构造函数创建一个对象并返回。单例模式在实践中要考虑多线程的问题。

实现

这里实现的单例模式有三个约定：

  • 不允许在创建实例时指定参数，否则初始化完成后下一次请求时使用不同参数就会有问题。如果实现通过根据参数请求访问同一个实例，可以用工厂模式。
  • 不允许被继承。如果需要一个单例的基类，而实际类型直到运行时才能确定，也可以用工厂模式。
  • 要实现延迟加载。

C#的实现

这里 C# 的实现有一些共同特点：

  • 一个私有的无参数的构造函数，既防止其他类实例化它，也防止被继承。
  • 类是 sealed 的。其实由于构造函数为私有无参数的，这不是必须的。但是可以帮助 JIT 来优化。
  • 一个静态的变量来引用被创建的单个实例。
  • 一个公共的方式来获取对这个对象的引用，或者在需要时创建它。

关于 beforefieldinit

上面的代码有一句被注释了的静态构造函数，当有这个静态函数时，或许会提升下这个方案的 lazy 程度，因为这样可以让编译器不在这个类上加 beforefieldinit 标记。加上这个标记后，CLR 只保证在该类的第一次访问前进行类初始化，但不能保证是“刚好”访问前，无法保证是延迟加载。

关于这个标记目前查到的资料是：

  • 这个标记存在于 CLI 规范 （ECMA 335）中，但在 C# 规范中没有提及，C# 编译器标记所有没有静态构造函数的类为 beforefieldinit，有静态构造函数的类则没有。
  • 在不同版本（比如1.1、2.0 或 4.0）之间以及不同模式（Debug 和 Release）之间，它对于类型构造器触发的时间都有可能有不同。
  • .NET 4.0 之前，一个有显式构造函数的类，确保可以在它的成员第一次被访问前立即被调用构造函数，不早也不晚，而没有显式构造函数的类，确保可以在它的成员第一次被访问的某个时间被调用构造函数（也许这样可以便于 CLR 进行优化？）。
  • .NET 4.0 之后，在 Release 模式下，即使类的静态方法被调用，只要被调用方法里不涉及它的静态字段，类型初始化器就不会触发，可以说是非常 lazy 了。

综上，我觉得还是不依赖于这个标记，只要按照类型初始化器只能保证在类型成员第一次被引用前被调用的原则来处理进行，至于延迟的那点时间其实并不重要。

取舍

实现一和实现三不建议使用，第一个线程不安全，第二个不好移植到 Java 还容易出错，即使正确了也不必其他几个有优势。

平时可以使用实现四，除非需要知道该单例是否已经实例化或怕涉及循环引用时，这种时候可以用实现二。实现五更优雅，但是比实现二、实现四麻烦，收益比较低。

对于实现四和五，有很多情况下完全的延迟加载并不重要，除非这个单例的实例化开销很大，或者在别处有些副作用，此时可以省略显式静态构造器（即 beforefieldinit 标志位）。这样可以提升性能，因为这样会允许 JIT 编译器做 singel check 来确保该类型被初始化并且接下来不再检查。不过总体而言，单例模式的性能提升没有那么明显，所以取舍还是看个人。

有时候需要在单例构造函数中做些可能会抛出异常，但不会对整个应用程序造成致命错误的操作。可能程序可以修复这些问题并想重试，此时如果用类型初始化器（静态类型方式）来创建单例就会有问题。虽然不同的运行时处理这种情况的方式不一样，但极少可以重新运行类型初始化器的。即使有的话，代码也无法在其他运行时保持一致。此时可以使用实现二，用一个简单的锁，每次都检查一次，若之前没成功可以再次尝试初始化。

考虑到现在大部分程序都是在 .NET 4 以上了，我还是比较喜欢直接用实现六，既简单，也实现了完全懒加载。

上述解决方案的其他问题

上述解决方案进化到最后，在日常使用中已经没有问题了，但是还有些非正常状态下的情况，要不要考虑见仁见智：

  • 反射：虽然通过私有构造函数加唯一入口的方式可以控制在常规调用中只有一个实例，但是如果使用反射，就可以构造多个实例了。
  • 序列化：还有一种打破现有约束的方式就是先获取一个实例，然后序列化，再通过反序列化就获得了另一个可用的实例。

Java 的实现

C# 中的方式都可以运用于 Java，不过很多 Java 开发者会倾向于使用枚举的特殊机制来作为更好的选择。

单例模式的缺点

尽管单例模式作为经典的23个模式之一，解决了一些问题，但是关于它是否是一个好的编程习惯还是有一些争议。主要争议点在于，首先它是个全局访问点，在很多人眼里全局访问点是个不好东西。其次，它将逻辑全部放在了自己内部，而不是通过依赖注入等抽象的方式，在很多人眼里这不利于测试。具体的讨论可以见这里和这里。不过我觉得这不是什么大不了的事，本来软件的设计就不是一个死板的东西，都是在平衡各方面的需要。

其他

单例模式和全局变量

单例模式的一个重要作用是作为一个全局访问点，它和全局变量的区别在于：

  • 全局变量会在程序一开始时就初始化，如果该对象非常耗费资源，这种方式有些浪费，因为并不是每次程序在执行过程中都会使用它。
  • 全局变量可以提供全局的访问，但不能确保只有一个实例。

单例模式和静态类或者静态属性

有时候可以用静态类或者静态属性来实现单例的效果，单例模式的优势在于：

  • 单例模式可以继承接口，方便测试和依赖注入。
  • 单例模式可以继承某个有用的基类（用的不多，而且基类如果有自己的构造方法会有隐患）。
  • 单例模式作为一个普通的对象，具有普通对象的一切特征，比如可以当做参数传入方法，而静态类则有很多限制。

总结

单件模式是一个很有意思的模式。一方面，它的应用很广泛，许多框架和业务逻辑都需要这样一个只实例化一个类的保证。另一方面，它并不像许多模式那样有固定的组织代码的方式，而是要充分利用各个语言的特点，来制定不同的实现方式。。

依赖反转、控制反转、依赖注入和策略模式

严格的说，标题里的几个东西并排在一起很奇怪，因为它们既不是并列关系也不是递进关系，甚至有些之间没有直接关系。但在我使用它们进行解耦时一开始确实没弄清楚它们之间的区别。在网络上搜索中文资料时讨论前三者的较多，关于依赖注入和策略模式的取舍鲜提及。在查阅了一些英文资料和 Stack Overflow 之后，我的疑问得到了解答，现在做一些总结。

本文的前提是你知道什么是依赖，什么是策略模式以及为什么很多情况下组合优于继承。这些都是非常基础的东西，网上有太多文章在反复阐述这些概念了，而我也没法给出更好更全面的解释。

依赖反转

依赖反转也可以翻译成依赖倒置，英文叫 Dependency Inversion。它是一个面向对象的设计原则，旨在解耦模块之间的关系。在这个原则下：

  • 高层模块不再依赖于低层模块，它们都依赖于抽象；
  • 抽象不依赖于具体细节，细节依赖于抽象。

这是意思呢？如果一个人没有学习过任何软件设计的技巧，他写出来的代码很大程度上是高层模块依赖于低层模块的。比方说你的代码中要使用一个类库，那么第一反应是实例化一个该类库的对象。在这里，类库是低层模块，你写的逻辑代码是高层模块，低层代码的作用就是给高层代码使用的。此时，你的逻辑代码就依赖于这个类库，一旦这个类库变了，那么你的代码逻辑很有可能也要跟着改变。从传统的软件开发方法上讲，这是一个很正常的面向对象设计，或者叫做代码复用设计，这样是没什么问题的。

但在别的场景下，这么做就有可能不合适了。比如你要写一个写日志的逻辑，允许用户自定义日志的方式，可以写进数据库，也可以写进文件中。可是此时用户代码还没有呢，你这边代码如何进行呢？显然大家都知道可以通过接口，委托之类的来实现。此时做到了第一步，既高层代码不依赖于低层代码的具体实现。比如：

Robert Cecil Martin 对于遵循该原则提出了三个启发性规则：

  • 任何变量都不应该持有一个指向具体类的引用；
  • 任何类都不应该从具体类派生；
  • 任何方法都不应该重写它的任何基类中的已经实现了的方法。

当然应该没有哪个系统真正完全做到这三点，这只是对于业务频繁改动的系统的一些关键模块设计上的参考。

我个人觉得对于依赖反转原则要注意三点，首先，在业务逻辑中可能会有变化的地方要应用该原则。其次，养成面向接口编程的习惯。最后减少用new来直接实例化对象，改为工厂或依赖注入等方式。

控制反转

控制反转这个概念从某种意义上说是依赖反转思想在框架开发上的体现。控制反转的英文叫 Inversion of control（简称 IOC）。从字面上来看它反转的不是依赖，而是控制，这里的控制是什么意思呢？还是要从早期的软件开发说起，早期的程序没有框架的说法，所有的控制都是程序编写者来编写。比如 MVC 的架构下，要自己通过控制器从视图上获取输入，调取模型进行处理，然后反馈给视图，这一切都通过手动编码来完成。后来有了框架，软件编写变得简单了。我们只要按照框架的接口绘制好视图，建好模型，编写好控制器处理程序即可，一切的控制都由框架来完成了。这就形成了反转，即控制从编写代码的人手上反转到了框架手里。

控制反转是一个很大的概念，因为控制既可以指控制流，也可以指控制哪个组件来做事，也许还有别的我一时想不出的控制。所以2004年 Martin Fowler 在他的一篇文章中结合当时流行的 IOC 容器，针对其中的一个部分——也是框架中非常重要的一个部分——控制组件，提出了一个新的概念，叫依赖注入。注意依赖注入不是控制组件的唯一方式，其他还有服务定位器（Service Locator）。不过大多数IOC容器的思想都是依赖注入。

异常和返回值

在开发过程中，经常要对一些可能出错的代码进行掌控，比如某个变量有可能为 null、调用资源时有可能会遇到问题以及调用类库时出现错误等等。平时常用的方法大概有三种：

  • 通过 if...else 语句对可能出现的情况进行判断，并通过返回值给出信息；
  • 抛出异常；
  • 使用断言。

下面说说这三种方式有什么不同。

断言

断言的作用是提示开发者，代码绝对不能进行到这里，如果进行到这里，一定是编码过程中出现了错误，需要开发者在开发阶段就要修正该错误。由于有些语言中断言在发布后就不会起作用，所以千万不能在用户和调用者可能出错的地方使用断言。

返回值

返回值和异常的选择应该是大多数人比较纠结的。先说返回值。在当初学习 C 语言时，教材上对于程序的出错不少都是使用返回一个整数来提醒，比较经典的是0表示正确，-1表示一种错误，-2表示另一种错误等等。这个方法看似简单，但实际使用中有以下问题：

  • 需要去维护一个错误代码表，不直观；
  • 在一些编程思想中会认为，函数的返回值就是表示整个函数产生的结果，返回出错代码违反了这个原则；
  • 如果某个函数实际调用了许多其他通过返回值来提示错误的函数，如何定义该函数的返回值则十分困难；
  • 代码中充斥了臃肿的 if ... else 语句来判断出错情况。

在上一家公司时，发现基本类库中喜欢使用 try ... catch 捕获异常，并通过自定义返回字符串的方式来提示，所以每次调用类库中的方法都需要判断返回空字符串表示调用正常，不然返回的就是出错的情况。这种方式看似很合理，但是有一个问题，就是在多层调用时，如果在多个函数中有错误，返回值的处理很麻烦，每一个间接层都要将新的错误信息拼接到返回值中，最后处理时要自己去判断每个错误发生在哪个函数中。

异常

异常算是面向对象语言专属的出错处理方式，在 C# 、Java、Python 以及一些面向对象语言中异常都在语言核心部分。异常一般会自己维护一个错误，一个记录出错信息的变量和一些其他辅助属性。通过错误栈可以很清楚的了解到这个错误发生在哪个函数中，并了解在调用该函数期间经过了哪些其他函数。错误信息则很清晰的表明当时发生的错误。另外，不少语言还可以通过继承内置的异常来自己创建异常类，增加了使用的灵活性。

异常和返回值相比，优点在于：

  • 它不需要改变函数的签名。返回值往往要将一个本身不需要返回任何值的函数改成返回整型或字符串类型，这种设计不是很好。
  • 它提供的信息要丰富。异常不光是告诉你错了，还能告诉你错在哪、什么原因、中途经过了哪些调用栈以及一些自定义的错误。相对而言返回值提供的信息很难做到这么丰富。
  • 不用像返回值的处理方式一样要记住每个函数的每个返回值代表什么意义，可以直接通过异常自带的信息来判断。
  • 可以通过在最上层放一个全局的异常处理来避免意想不到的出错，而返回值则做不到这点。

异常和返回值相比，缺点在于：

  • 异常的性能一般要比返回值差。

选择异常和返回值的一些争论

关于异常和返回值的讨论网上有许多不同的意见，争论多的原因不外乎大家对于代码设计的哲学不同。

有种说法是预期会出错的地方用返回值，预期不会出错的地方用异常，比如《程序员修炼之道》中对异常的看法是，如果它有可能出错，应该用返回值；如果它不应该出错，应该用异常。但我觉得这只解决了关于返回值会有遗忘的情况，对于破坏了函数签名的问题没有解决，所以我不是很赞同这种设计。

对于我常使用的 C#，微软设计了一整套好用的异常体系，在许多类库里对于传入参数的检测都用异常来抛出，看上去是站在异常这一派的。但是这几年用微软 TryXXX 的方式越来越多，发现配合 out 参数也是一种能保证函数签名完整性的方式。Go 的 error 机制也类似于这样。

我的选择

首先如果加入的团队已经有方案了，那肯定是遵循团队方案。如果是自己负责架构设计，则更倾向于尽量使用异常机制来保证方法签名和代码逻辑流的独立性。所以在大部分情况下遵循以下几点：

  • 底层类库永远不要捕获异常，有异常让它直接抛出，让调用方去处理。
  • 中间层业务代码中，除非能保证处理异常的代码在业务层面完全纠正异常的行为，否则还是让它抛出，让调用它的最高层统一处理。什么叫完全纠正异常呢，比如要调用个网络连接，但是第一次失败了，不过本身就设定了三次重试，前两次失败的异常就自己抓住，到第三次再抛出个三次调用失败的异常出来给调用方。
  • 中间层业务代码还有一种情况就是本身是作为服务提供的（其实这从某种意义上说算是最高层了），那么可以统一定义返回格式，将异常处理完之后返回状态码。
  • 最高层代码中，捕获的异常一般会记录日志然后统一处理，比如 Web 系统跳转到一个统一的错误页。
  • 最高层代码中，对于可能预测到的用户的主动行为，考虑到性能问题，在代码中对常见错误进行判断并返回相应值。

另外有几种情况，有可能会根据具体使用情况来设计的：

  • 该函数本身就应该返回错误，比如 bool IsSuccess() {do someting...}，成功返回 true，错误返回 false，则可以不在错误时抛出异常。而有些人觉得这样函数的签名就不该设计为 bool，而是 void DoSomething(){do someting...}，因为这个函数实际是没有产生输出的。这就看具体实现者的理解了。
  • 该函数预期会得到多种结果，其中一些结果在用户角度看是错误的，但是从程序角度看不是。比如校验函数 string validate (Model model)，返回对于传入参数的校验结果；或登录函数 string Login()，可以返回“用户名错误”或“密码错误”等多种情况。不过这种情况下，我也不喜欢使用返回值而破坏签名，更喜欢设计成 bool validate (Model model, out message) 这样的方式。
