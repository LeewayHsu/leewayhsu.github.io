严格的说，标题里的几个东西并排在一起很奇怪，因为它们既不是并列关系也不是递进关系，甚至有些之间没有直接关系。但在我使用它们进行解耦时一开始确实没弄清楚它们之间的区别。在网络上搜索中文资料时讨论前三者的较多，关于依赖注入和策略模式的取舍鲜提及。在查阅了一些英文资料和 Stack Overflow 之后，我的疑问得到了解答，现在做一些总结。

本文的前提是你知道什么是依赖，什么是策略模式以及为什么很多情况下组合优于继承。这些都是非常基础的东西，网上有太多文章在反复阐述这些概念了，而我也没法给出更好更全面的解释。

## 依赖反转
依赖反转也可以翻译成依赖倒置，英文叫 Dependency Inversion。它是一个面向对象的设计原则，旨在解耦模块之间的关系。在这个原则下：
- 高层模块不再依赖于低层模块，它们都依赖于抽象；
- 抽象不依赖于具体细节，细节依赖于抽象。

这是意思呢？如果一个人没有学习过任何软件设计的技巧，他写出来的代码很大程度上是高层模块依赖于低层模块的。比方说你的代码中要使用一个类库，那么第一反应是实例化一个该类库的对象。在这里，类库是低层模块，你写的逻辑代码是高层模块，低层代码的作用就是给高层代码使用的。此时，你的逻辑代码就依赖于这个类库，一旦这个类库变了，那么你的代码逻辑很有可能也要跟着改变。从传统的软件开发方法上讲，这是一个很正常的面向对象设计，或者叫做代码复用设计，这样是没什么问题的。

但在别的场景下，这么做就有可能不合适了。比如你要写一个写日志的逻辑，允许用户自定义日志的方式，可以写进数据库，也可以写进文件中。可是此时用户代码还没有呢，你这边代码如何进行呢？显然大家都知道可以通过接口，委托之类的来实现。此时做到了第一步，既高层代码不依赖于低层代码的具体实现。比如：

````
public void Log(string msg)
{
    ILog loger = new DBLoger();//DBLoger如何实现与写日志的逻辑无关了
    loger.Log(msg);
    ...
}
````

可是如果只是定义了一个接口给低层，在高层中还需要实例化具体实现的话，还是会有依赖。如果想要将日志写进文件时，就需要修改逻辑代码了。此时需要将接口的具体实例化通过别的方式来进行，比如简单工厂：

````
public void Log(string msg, LogerType logerType)
{
    ILog loger = InitLoger(logerType);//根据传入的类型来决定是返回DBLoger还是FileLoger
    loger.Log(msg);
    ...
}
````

这样一来只要接口不变，无论是添加新的日志方式，还是修改原有的日志实现，这段代码都不需要进行任何改动。因为此时高层代码（写日志逻辑）和低层代码（日志实现）已经解耦了。这里的简单工厂可以用其他方式代替，比如查找表之类的。

Robert Cecil Martin 对于遵循该原则提出了三个启发性规则：
- 任何变量都不应该持有一个指向具体类的引用；
- 任何类都不应该从具体类派生；
- 任何方法都不应该重写它的任何基类中的已经实现了的方法。

当然应该没有哪个系统真正完全做到这三点，这只是对于业务频繁改动的系统的一些关键模块设计上的参考。

我个人觉得对于依赖反转原则要注意三点，首先，在业务逻辑中可能会有变化的地方要应用该原则。其次，养成面向接口编程的习惯。最后减少用`new`来直接实例化对象，改为工厂或依赖注入等方式。

## 控制反转
控制反转这个概念从某种意义上说是依赖反转思想在框架开发上的体现。控制反转的英文叫 Inversion of control（简称 IOC）。从字面上来看它反转的不是依赖，而是控制，这里的控制是什么意思呢？还是要从早期的软件开发说起，早期的程序没有框架的说法，所有的控制都是程序编写者来编写。比如 MVC 的架构下，要自己通过控制器从视图上获取输入，调取模型进行处理，然后反馈给视图，这一切都通过手动编码来完成。后来有了框架，软件编写变得简单了。我们只要按照框架的接口绘制好视图，建好模型，编写好控制器处理程序即可，一切的控制都由框架来完成了。这就形成了反转，即控制从编写代码的人手上反转到了框架手里。

控制反转是一个很大的概念，因为控制既可以指控制流，也可以指控制哪个组件来做事，也许还有别的我一时想不出的控制。所以2004年 Martin Fowler 在他的[一篇文章](https://www.martinfowler.com/articles/injection.html)中结合当时流行的 IOC 容器，针对其中的一个部分——也是框架中非常重要的一个部分——控制组件，提出了一个新的概念，叫依赖注入。注意依赖注入不是控制组件的唯一方式，其他还有服务定位器（Service Locator）。不过大多数IOC容器的思想都是依赖注入。

## 依赖注入
使用依赖注入的前提是想要注入到框架中的功能由一个个组件的接口组成，通过依赖注入来将具体的接口实现注入到框架中去。有三种主要的依赖注入方式：
- 构造器注入
- 设置器注入
- 接口注入

构造器注入和设置器注入的本质是一样的，都是在类中定义一个接口，在运行时通过函数传参的方式传入具体接口实现对象，区别在于一个是通过构造函数，一个是通过专门的设置函数，下面的例子中列出了这两种方式：

````
public class LogManager
{
    ILog _loger;

    public void Log(string msg)
    {
        _loger.Log(msg);
    }

    //构造器注入
    public LogManager(ILog loger)
    {
        _loger = loger;
    }
    //设置器注入
    public void SetLoger(ILog loger)
    {
        _loger = loger;
    }
}
````

接口注入我很少使用，也没看明白这么做的原因，就是为了给实现注入的类一个接口约束？下面是代码：

````
public interface ILogerManager
{
    void InjectLoger(ILoger loger);
}

public class LogerManager:ILogerManager
{
    ILoger _loger;

    public void InjectLoger(ILoger loger)
    {
        _loger = loger;
    }

    public void Log(string msg)
    {
        _loger.Log(msg);
    }
}
````

## 依赖注入和策略模式之间的关系
以上的部分其实在中文技术博客中是很常见的，不过我在真实项目中自己重构的时候则遇到这样的疑问，除却IOC容器的使用外，如果自己在编码中注入依赖，和使用策略模式有什么区别。这个疑问没找到中文的文章，还好在 Stack Overflow 上找到了解答。

依赖注入和策略模式本质上都是通过接口来解耦，二者的主要区别要实现的目的上。如果需要创建一个在运行时固定的对象，策略模式和依赖注入几乎效果是一样的。当对象的依赖关系会在运行时会发生变化时，一般使用策略模式。运行时不会发生变化的情况，比如向框架中注入接口实现，则更多的使用依赖注入。另外，在单元测试中一般用依赖注入的情况更多一点。
