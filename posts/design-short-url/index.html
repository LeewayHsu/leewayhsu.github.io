设计短链接

短网址的实现原理是通过建立一个短地址和长地址的一一映射关系，然后短链服务器在接受请求时通过 301 或 302 重定向到长地址来实现（一般用临时重定向302）。

1. Hash 算法

很容易想到的就是通过 hash 将一个长字符串转到一个固定的短字符串。这里不需要 MD5 和 SHA 这些复杂的哈希算法，因为不需要考虑反向解密的难度，所以只需要关心哈希算法的计算速度和冲突概率。比较著名的一个哈希算法是 MurmurHash 算法。

MurmurHash 算法提供了两种长度的哈希值，一种是 32bits，一种是 128bits，为了让最终生成的短网址尽可能短可以选择 32bits 的哈希值。

这里选择 Hash 得到一个数字而非字符串，然后通过62进制（0～9、a～z、A～Z）转成字符，进一步缩短大小。

解决冲突

Hash 的话带来的一个问题就是解决冲突。

生成短地址后，为了能反向查长地址，肯定要将二者的映射关系存起来。如果使用 MySQL 的话，可以在短地址字段上加唯一索引。

  • 插入时如果没有该短地址就成功
  • 如果有的话先取出来对比下是不是当前的原始地址，是的话直接使用。
  • 不是的话：
    • 可以在短地址后面加1、2、3等
    • 可以在短地址后面加特殊字符（比如 [DUPLICATED]）后再计算哈希值。

如果为了节省唯一索引的开销，还可以用布隆过滤器，把短地址都加进去。这样只要生成的短地址不在过滤器中，就一定未使用。

2. 发号器

发号器就是用自增ID来生成地址，然后用62进制转成对应的字符形式。发号器在工程上可能更适用一点，因为相对比较简单，而且具有唯一性。

小型系统可以用 MySQL 的自增键作为发号器。大型系统可以用 Redis 来做。再复杂点的话可以使用一些生成唯一ID的中间件，比如美团的 Leaf。

考虑并发的话，可以简单用分开发号的形式。比如实现1000个逻辑发号器，分别发尾号为0到999的号。每发一个号，每个发号器加1000，而不是加1。并发量再大的话就将逻辑分开变成物理分开即可。

对于 MySQL 还可以通过提前取一批号的方式，然后在内存里用原子计数器来新增。

发号器的一个争议在于会出现同一个地址来了多次会得到多个短链的情况。首先这种情况是可以接受的。其次如果不能接受，就要再建一个长链和短链的映射表来进行判重，其实意义不大，还影响并发。一个折中的方案就是只存近一小时的数据，就可以解决大部分问题。
