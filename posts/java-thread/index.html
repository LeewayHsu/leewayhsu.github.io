Java 并发——Thread 总结

1. Thread 、Runable 和 Callable

Java 中的 Thread 类表示一个线程。它实现了 Runnable 接口，该接口是个 @FunctionalInterface，仅有一个 public abstract void run() 方法。有两种方式构建一个 Thread 类：

  • 实现 Runnable 接口，并将其实现传入一个 Thread 类；
  • 继承 Thread 类，覆写其中的 run() 方法。

推荐第一种方式，这样具体的任务可以和 Thread 类解耦。

使用 Thread 执行任务时，一般使用 start() 方法，这样会开启一个新线程来执行该任务。如果直接使用 run() 方法，就会在当前线程执行，一般不会这么做，这么做跟直接执行方法没什么区别。

Callable<V> 接口类似 Runnable，但它返回一个值。它表示一个异步计算，并最终返回一个类型为 V 的对象。它也是个 @FunctionalInterface，仅有一个 V call() 方法。

2. Future 和 FutureTask

Future<V> 接口持有一个异步计算的结果。它的作用是，当开启一个计算任务时，将 Future 对象传入，当它准备就绪的时候就可以从其中获得结果。有以下方法：

public interface Future<V> {
    V get();
    V get(long timeout, TimeUnit unit);
    void cancel(boolean mayInterrupt);
    boolean isCancelled();
    boolean isDone();
}
  • 第一个 get 方法会阻塞直到计算完成。
  • 第二个 get 方法如果超时了会抛出 TimeoutException。
  • 如果执行计算的线程被 interrupt 了，上面两个方法都会抛出 InterruptedException。如果计算已经完成了，get 立刻返回。
  • isDone 方法在计算完成时返回 true，否则返回 false。
  • cancel 方法可以取消计算：
    • 如果计算还是开始，就会取消并且不会再执行了；
    • 如果计算已经开始，如果 mayInterrupt 参数为 true 的话，线程会被 interrupt。

上面的 Future<V> 只是一个接口，代表了一种规范。Java 中有一个对这个接口的实现，就是 FutureTask 类。

FutureTask 类不光继承了 Future<V> 接口，还继承了 Runnable 接口（它实现了 RunnableFuture 接口，后者继承了 Runnable 和 Future<V> 接口）。所以也可以将它看成是 Future 和 Runnable 的 wrapper。它主要是接收一个 Callable 并将其结果以 Future 的方式返回。

一个使用 FutureTask 的例子：

Callable<Integer> myComputation = . . .;
FutureTask<Integer> task = new FutureTask<Integer>(myComputation);

Thread t = new Thread(task); // it's a Runnable
t.start();

. . .

Integer result = task.get(); // it's a Future

3. 中断一个线程

Java 的早期版本有一个 stop 方法可以让其他线程终止一个线程，不过已经 deprecated 了。现在已经没有方式强制一个线程终止了。

只能用 interrupt 方法来请求中断一个线程，调用 interrupt 方法时，设置了该线程的 interrupted 状态，这是个 boolean 标识，存在于每个线程中。

线程感知到被中断后，可以选择忽略也可以选择保存好状态退出等等，都由开发者自己决定。

有两种机制来让线程感知中断：

  • 一种是如果线程正在调用会抛出 InterruptedException 的方法，比如 sleep 或 wait，此时线程会捕获到这个异常；
  • 一种是线程自己主动查询它是否被中断，可以调用 Thread.currentThread 获取当前线程，然后调用 isInterrupted 方法看是否返回 true。

检查线程是否被中断有两个相似的方法：

  • 实例方法 isInterrupted，用于检查该线程对象是否被中断，它不会清除线程的中断状态；
  • 静态方法 interrupted，用于检查当前线程是否被中断，它会清除线程的中断状态。

其实二者都是调用 private native boolean isInterrupted(boolean ClearInterrupted)：

  • 第一个方法调用 this.isInterrupted(false)
  • 第二个方法调用 currentThread().isInterrupted(true)

关于中断信息的捕获和处理，目前没有官方的最佳实践。我个人的实践是，不要在低层的代码中捕获 InterruptedException 然后忽略它。如果不知道如何处理该异常，有两个合理的选择：

  • 在 catch 子句中，调用 Thread.currentThread().interrupt() 来设置 interrupted 状态，这样 caller 可以测试这个状态；
  • 更好的选择是将方法标记为 throws InterruptedException，然后方法内不捕获该异常，这样上层可以捕获它。

4. 线程状态转换

Java 语言定义了6种线程状态：

  • New
  • Runnable
  • Blocked
  • Waiting
  • Timed waiting
  • Terminated

可以通过调用 getState 方法来获取线程状态。

在任意一个时间点中，一个线程只能有且只有其中的一种状态，并且可以通过特定的方法在不同状态之间转换：

  • 新建（New）：创建后尚未启动的线程处于这种状态。
  • 运行（Runnable）：包括操作系统线程状态中的 Running 和 Ready，也就是处于此状态的线程有可能正在执行，也有可能正在等待着操作系统为它分配执行时间。
  • 无限期等待（Waiting）：处于这种状态的线程不会被分配处理器执行时间，它们要等待被其他线程显式唤醒。以下方法会让线程陷入无限期的等待状态：
    • 没有设置 timeout 参数的 Object::wait() 方法；
    • 没有设置 timeout 参数的 Thread::join() 方法；
    • LockSupport::park() 方法。
  • 限期等待（Timed Waiting）：处于这种状态的线程也不会被分配处理器执行时间，不过无须等待被其他线程显式唤醒，在一定时间之后它们会由系统自动唤醒。以下方法会让线程进入限期等待状态：
    • Thread::sleep() 方法；
    • 设置了 timeout 参数的 Object::wait() 方法；
    • 设置了 timeout 参数的 Thread::join() 方法；
    • LockSupport::parkNanos() 方法；
    • LockSupport::parkUntil() 方法。
  • 阻塞（Blocked）：线程被阻塞了，“阻塞状态”与“等待状态”的区别是“阻塞状态”在等待着获取到一个排它锁，这个事件将在另外一个线程放弃这个锁的时候发生；而“等待状态”则是在等待一段时间，或者唤醒动作的发生。在程序等待进入同步区域的时候，线程将进入这种状态。
  • 结束（Terminated）：已终止线程的线程状态，线程已经结束执行。正常 return 和抛出异常结束的线程应该都是这个状态

4.1 关于阻塞

日常开发中，对于线程的状态最关心的就是阻塞。这里的阻塞是一个比较广泛的词，指的是线程释放了 CPU 资源并处于一个等待运行的状态。对于阻塞而言：

  • Blocking IO 中等待时，其实还是 RUNNABLE 状态；
  • wait、park、sleep 都是 WAITING 状态，不过分有限期和限制；
  • 被 synchronized 阻塞是进入 BLOCKED 状态；

有一道莫名其妙的却很流行的面试题是问 sleep 是否会释放锁。sleep 是一个和锁无关的操作，自然也不存在释放锁的动作。真正和锁有关的是 wait，因为 wait 必须在 synchronized 块中调用，所以 wait 必须是持有锁的线程释放锁然后等待 notify。

至于还有人问过 park 是否释放锁，park 本来就不一定有锁，何谈是否放弃锁。park 的逻辑应该是用了 premit 值，然后调用虚拟机的方法让线程变成 WAITING。

5. Daemon 线程

daemon 线程是一种特殊的线程，它和普通线程的区别在于：如果只有 daemon 线程，虚拟机会退出（反过来说只要还有非 daemon 线程在执行没退出，虚拟机就不会退出。所以如果要用线程来维持程序持续运行的话不能将它设置为 daemon 线程）。

这样将一个线程变成 daemon thread：

t.setDaemon(true);

daemon 线程一般是服务其他线程用的，比如计时器。

6. 处理未捕获异常

一个线程的 run 方法不能抛出任何 checked 异常，但它可以被一个 unchecked 异常终止，这种情况下该线程会死亡。

在 Thread 的 start 外面是没法捕获到线程中的未捕获的异常的，比如：

public static void main(String[] args) {
    try {
        new Thread(() -> { }).start();
    } catch (Exception e) {
        // 无法捕获
        System.out.println(e.getMessage());
    }
}

当然 start 内部可以捕获：

public static void main(String[] args) {
    new Thread(() -> {
        try {
        } catch (Exception e) {
            System.out.println(e.getMessage());
        }
    }).start();
}

未捕获的异常会被传入一个 handler。如果想在外面控制的话，可以给该线程注册一个 handler。该 handler 需要实现 Thread.UncaughtExceptionHandler 接口：

@FunctionalInterface  
public interface UncaughtExceptionHandler {
    void uncaughtException(Thread t, Throwable e);  
}

可以通过 setUncaughtExceptionHandler 实例方法将 handler install 入任何线程：

Thread thread = new Thread();
thread.setUncaughtExceptionHandler(new CustomerUncaughtExceptionHandler());
thread.start();

Thread 类有一个类级别的默认 UncaughtExceptionHandler，初始状态下为 null，不过可以通过 Thread 类的 setDefaultUncaughtExceptionHandler 方法来设置。设置完了之后它会对所有线程有效：

Thread.setDefaultUncaughtExceptionHandler(new CustomerUncaughtExceptionHandler());

当出现未处理异常时，会先检查 UncaughtExceptionHandler，如果为 null 则检查 DefaultUncaughtExceptionHandler。

如果以上两个都没设置，线程则使用它的 ThreadGroup 作为 UncaughtExceptionHandler。ThreadGroup 是用于管理一组线程的一个类。它本身实现了 Thread.UncaughtExceptionHandler 接口。它的 uncaughtException 方法的逻辑如下：

  1. 如果 thread group 有 parent，则调用 parent 的 uncaughtException 方法；
  2. 否则，如果 Thread.getDefaultUncaughtExceptionHandler 方法返回一个非 null 的 handler，则使用它；
  3. 否则，如果 Throwable is an instance of ThreadDeath，不会发生任何事；
  4. 否则，线程的名字和 Throwable 的 stack trace 被打印进 System.err。

7. 线程的开销

JDK 8 下，Java 的线程大概需要1M的空间（据说 JDK  11 版本优化到了 40 KB），因此如果有 BUG 导致不停新开线程的话，会导致 OOM。
